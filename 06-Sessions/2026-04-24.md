### 1.1.7 funnel_edges / edge_type Latent Bug

`funnel_edges.edge_type` column is `text` type — NOT an enum. But `commit_discovery_result` RPC contains:
```sql
(v_edge->>'edge_type')::edge_type
```

PostgreSQL interprets `::edge_type` as casting to a type named `edge_type`. Since no such type exists, this cast will fail at runtime with "type edge_type does not exist" the first time any funnel edges are committed. This code path has never been hit (funnel_edges=0 rows, and Gemini has returned no funnel edges in test runs). **Latent crash.**

## 1.2 Pipeline Code

### 1.2.1 discover_creator.py

**Schema column references:**
- `is_primary`: present in `ProposedAccount` Pydantic model and passed to `commit_discovery_result` RPC ✓ (column now exists in DB)
- No direct table column references — all writes go through the RPC ✓

**Enum value handling:**
- `monetization_model`: `_normalize_monetization_model()` exists, called before commit ✓
- `platform`: Values from Gemini are injected via `_PLATFORM_ENUM` in the schema construction. `_PLATFORM_ENUM` is MISSING `patreon` compared to live DB — minor (Gemini wouldn't realistically map a creator to patreon as primary, but gap exists).
- No validation of `platform` values from Gemini before they're passed to `commit_discovery_result`. If Gemini returns a value not in `_PLATFORM_ENUM`, the Pydantic model accepts it as a plain string, and the RPC cast `(v_account->>'platform')::platform` would crash at DB level.
- `account_type`: Validated by Pydantic Literal type ✓
- `edge_type`: Passed as plain string, no validation. When funnel edges land: latent crash (see §1.1.7).

**Error handling:**
- `commit()` raises on failure → caught by `run()` → calls `mark_discovery_failed()` ✓
- IF `mark_discovery_failed()` also throws: exception is caught and printed to console only. Nothing persists. Creator remains in previous state. Silent failure mode.
- No retry logic around `mark_discovery_failed()` — it can silently fail.
- Worker (`worker.py`): `asyncio.gather(*futures, return_exceptions=True)` swallows exceptions from individual discovery tasks. Errors are not logged to the console or DB if gather absorbs them.

**fetch_input_context():** Uses `httpx.get()` to scrape Instagram/TikTok URLs. Instagram returns a login redirect. Gemini receives garbage (login page HTML). Discovery "succeeds" but finds no real data. **This is the documented known limitation in PROJECT_STATE §20.** Consequence: all 3 creators were likely discovered from garbage context, relying almost entirely on Gemini's prior knowledge of the handles.

**No content pipeline invocation:** `run()` calls `scrape_instagram_profile()` after commit, but no calls to `flag_outliers()` or `refresh_profile_score()` anywhere.

### 1.2.2 apify_scraper.py

**flag_outliers not called:** After upserting posts into `scraped_content`, `flag_outliers()` is never called. Outlier detection is permanently offline.

**refresh_profile_score not called:** Same — never invoked after scraping.

**Error handling:** Exceptions are only console-logged. No status update to the creator or profile. If an Apify run fails mid-scrape, there is no record of the failure in Supabase.

**view_count=0 for images:** `view_count = item.get("videoViewCount", 0) or 0` — correct for images (they have no videoViewCount). But outlier detection using view_count across all post types will produce skewed medians.

**Profile insert fallback (line 39-44):** Fallback path creates a profile WITHOUT `is_primary`. If called for a creator whose profile was already created by `commit_discovery_result`, this would conflict on `(workspace_id, platform, handle)`. The upsert would need an `on_conflict` clause — but this path uses `.insert()`, not `.upsert()`. This would fail with a unique constraint violation if the profile already exists. The except clause catches this and logs it. Silent failure.

### 1.2.3 worker.py

- Polls `discovery_runs` for `status='pending'`, claims by updating to `'processing'` ✓
- No claim race-condition protection at scale (noted in comments) — acceptable for current usage
- Does not invoke content scoring or outlier detection
- `asyncio.gather(return_exceptions=True)` swallows per-task exceptions — errors disappear silently

### What We Did

### Key Decisions

- **Rule cascade over weighted-sum identity scoring** — "shared OnlyFans URL" is categorically 1.0 proof, not 0.3 evidence. Rule cascade is auditable (each merge has a reason string in `evidence`), debuggable, and doesn't need ML infrastructure to calibrate. Research-agent critique convinced me the weighted-sum math was incoherent (weights summing to 1.2 thresholded at 1.0 is arbitrary).
- **CLIP over pHash for avatar similarity** — creators' avatars are adversarial for pHash (goth silhouettes, aesthetic shots, stock portraits). CLIP ViT-B/32 handles faces AND silhouettes AND product photography. 150MB server-side one-time download is a non-issue; 3-8% FP rate of pHash is fatal for a merge-candidate system where FP cost > FN cost.
- **Two-stage resolver, not BFS** — aggregator children become terminal leaves (no chaining). Budget is Apify $, not URL count. Asset-cap-by-count would silently truncate monetization data (exactly the most valuable output).
- **Gemini narrowed** — Gemini no longer classifies URLs or proposes accounts. It does canonicalization + niche + text_mention extraction only. URL classification owned by the deterministic classifier (rule gazetteer + cached LLM fallback).
- **Cross-workspace dedup runs on every commit**, not just within a bulk import. `profile_destination_links` is persistent per-workspace. Late-arriving evidence merges into existing creators automatically.
- **Manual Add Account triggers resolver expansion** with canonical-field protection (source='manual_add' branch in the RPC union-merges `known_usernames` only, doesn't overwrite human-confirmed identity).
