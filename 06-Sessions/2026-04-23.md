## Decisions Locked In Today

### Architecture
- **Archetype and vibe move to creator-level** (not content-level). Phase 2 migration removes `archetype` and `vibe` from `content_analysis`, adds them to `creators`. Rationale: these describe brand identity, which emerges across a full body of work, not from a single post.
- **Personas within a creator handled via `tags[]` on profiles** — no new schema. If a creator runs fitness + fashion niches, tag the accounts accordingly. Can always add a `persona` column later if needed; don't build it now.
- **Trends and audio get ONE new table** (`trends`) not three. Apify already returns audio metadata as text (`musicInfo` on IG, `musicMeta` on TikTok) — no audio fingerprinting needed. Normalize artist+title to an `audio_signature`, dedup into `trends`. Link content via `scraped_content.trend_id` column.
- **Creator niches reuse the existing `content_labels` table** by extending `label_type` enum with `creator_niche`. One new junction table (`creator_label_assignments`) mirrors existing content pattern. No duplicate taxonomy infrastructure.

### Pipeline choices
- **LLM routing strategy:**
  - Gemini 1.5 Pro → creator discovery, content visual analysis (multimodal)
  - Gemini Flash → quick classification with constrained enum outputs
  - Claude Sonnet → brand analysis, hook pattern analysis, narrative reasoning
  - Claude Opus/Sonnet → multi-step agent workflows when needed
- **Agents vs code:** everything stays as straight code for Phases 1–2. Claude agents introduced starting Phase 3 where iterative reasoning matters — Brand Analysis is the first real candidate, followed by label deduplication.

### Outlier detection locked in
- Formula: `current_views / median(last_N_posts)`
- N = last **50 posts** or last **90 days**, whichever smaller
- Minimum **15 posts** required before flagging activates
- Post must be ≥ **48 hours old**
- Flag threshold: multiplier ≥ **3.0**
- Skipping view-to-follower ratio — breaks for new/boosted creators
- Using median, not mean — viral hits don't poison the baseline

### Key Decisions
- **No pagination** on the accounts page — fetch all, filter client-side. Revisit at 500+ profiles.
- **Sort default**: quality score desc, unscored profiles (no `profile_scores` row) appear at bottom with "Not ranked" greyed-out state.

- **Archetype/vibe dropped from `AccountCard`** for this pass — Phase 2 moves them to creator level; building derivation logic now would be wasted work.
- **`medianViews`**: pulls from `profile_metrics_snapshots` (latest by date), falls back to live median computed over `scraped_content` last 90 days, falls back to "—".
- **`fanArchetype` removed** from `AccountCard` entirely — was cruft from earlier mock, not in schema.
- **Rank chips**: local client state only (not URL). Tracking type → URL (`?type=managed`). Rationale: tracking type is navigational/shareable; rank is a transient drill-down.
- **`account_type = 'social'` filter**: Instagram accounts page shows only social profiles. Monetization/link_in_bio/messaging accounts stay in the creator deep-dive Network tab.

### Architecture Pattern
Server Component (`page.tsx`) fetches: profiles + profile_scores (one query), profile_metrics_snapshots, scraped_content stats — all in parallel after getting profile IDs. Passes typed `AccountRowData[]` to `InstagramAccountsClient` (client component) which owns all filter/sort/search/stat logic.
