### What We Did

#### Always-on worker (Phase A)
- New `scripts/worker_ctl.sh` (committed) — generates `~/Library/LaunchAgents/com.thehub.worker.plist` with `RunAtLoad + KeepAlive + ThrottleInterval=10s`, logs to `~/Library/Logs/the-hub-worker.{log,err.log}`. Subcommands: `install / start / stop / restart / unload / status / log / err / uninstall`. The `restart` subcommand is the canonical way to make launchd respawn the worker with fresh code after pipeline edits.
- Plist generation bakes in absolute paths from `$(pwd)` and `$(command -v python3)` so the script is portable to other machines.
- Loaded the agent during the session — verified the worker auto-picks up pending runs within ~30s of being created.

#### Live progress UI (Phase B)
- Migration `20260425010000_discovery_runs_progress` adds `progress_pct smallint NOT NULL DEFAULT 0` + `progress_label text` to `discovery_runs`.
- Pipeline instrumentation: 5 emit points — 10% Fetching profile (resolver entry) → 35% Resolving links (after Stage A) → 70% Analyzing (before Gemini) → 90% Saving (in `discover_creator.run` before `_commit_v2`) → 100% Done. Implemented via a `progress` callback param threaded through `resolve_seed`, with a non-fatal `_make_progress_writer` wrapping the supabase update so a failed progress write doesn't blow up the run.
- New `getDiscoveryProgress` server action (next to the other discovery actions in `creators/actions.ts`).
- New `<DiscoveryProgress runId={...}>` client component (`src/components/creators/DiscoveryProgress.tsx`) — polls every 3s while a card is in `processing`, renders a thin progress bar + label + `{pct}%`, calls `router.refresh()` on terminal status flip. Polling chosen over Realtime — no `ALTER PUBLICATION` needed, only fires while in flight, simple.
- Drops into both the CreatorCard processing branch and the creator HQ "Discovering…" banner. `last_discovery_run_id` threaded through from `creators` query → `getCreatorsForWorkspace` → `CreatorCard` props.

#### Discovery surface bug sweep (Phase C)
- **`retry_creator_discovery` now updates `creators.last_discovery_run_id`** (migration `20260425020000`). Without this the progress UI polled the previous failed run forever, saw its terminal status, and `router.refresh()` looped without ever observing the new run. Worker actually ran the new attempt; UI just never saw it.
- **`bulk_import_creator` missing `::platform` cast** on the `discovery_runs` INSERT (migration `20260425030000`). Same shape as the retry RPC bug from earlier the same day — missed in that sweep because the RPC's `creators` and `profiles` INSERTs already cast. Visible: every Bulk Paste / Single Handle import errored with Postgres `22P02` in the toast.
- **Seed/primary profile URL was never written.** `_commit_v2` built the seed `accounts` entry with `url=None`; the upsert COALESCE preserved the null; the AccountRow rendered the seed handle as plain text instead of a clickable link. New `_seed_profile_url(platform, handle)` helper handles per-platform conventions (tiktok/youtube `@`-prefix, linkedin `/in/`, etc.). Backfilled Kira directly via SQL — the other 3 creators had URLs from earlier import paths.
- **Novel-platform persistence.** `_commit_v2` now appends a stub `profiles` entry for any `DiscoveredUrl` not already covered by `enriched_contexts`. Without this, novel platforms (Wattpad / Substack / etc.) — and `link_in_bio` aggregator parents that the resolver doesn't enrich — landed only in `profile_destination_links` and never rendered on the HQ page. Tenacity-wrapped `_call_actor` in `instagram.py` and `tiktok.py` retries 3× with 3-15s exponential backoff. `EmptyDatasetError` stays non-retriable.

#### Operational / debugging
- During smoke, hit a different bug: the launchd worker kept failing with `ApifyApiError: User was not found or authentication token is not valid` while my direct Python tests succeeded. Traced via `pgrep` + `ps eww` + a clean-env subprocess test — turned out `scripts/.env` held a revoked `APIFY_TOKEN` while Simon's shell had the working one (both 46-char, both prefixed `apify_ap`, different values). Launchd workers see only their plist's clean env, so they got the bad token. Fix: backed up `scripts/.env` → `.env.bak.{ts}`, replaced just the `APIFY_TOKEN=` line via Python rewrite (sed quoting was unsafe with token chars), restarted worker. Aria Swan's discovery then completed cleanly to 100% with 704K-follower IG + secondary IG `ariaaswan` (her "main" handle) + a `platform=other` stub via the new novel-platform persistence path.
- Verified the progress bar end-to-end via a synthetic SQL flip — bumped esmae's run to `progress_pct=42, progress_label='Fetching profile'` then to `70/Analyzing`; UI auto-advanced within one 3s poll cycle. No page reload.

### Key Decisions (sync 13)

- **launchd over Docker / systemd / brew services.** Native to macOS, zero deps, ships with the OS. ThrottleInterval=10s prevents tight crash loops if the worker ever has bad env vars.
- **Polling at 3s, not Supabase Realtime.** Realtime needs `ALTER PUBLICATION` per table + extra client setup. Polling at 3s is dead simple, only fires while a card is actually in `processing` state (component unmounts otherwise), latency is invisible at human pace.
- **5 progress stages, 2-3 word labels** ("Fetching profile" / "Resolving links" / "Analyzing" / "Saving" / "Done"). Per Simon's "few words, simple, don't overcrowd UI".
- **No ETA in v1.** Pipeline time varies a lot with destination count; a fake countdown would lie. The label tells you what's happening.
- **Progress writes are non-fatal.** Wrapped in try/except — if the supabase update fails, the pipeline keeps running.
- **Token rotation handled inline, not via a "rotate token" workflow.** Single line replacement in `.env` with backup; reversible if anything went wrong.
- **Three small commits per phase rather than one big batch.** 10 commits this pass. Easier `git log` reading; per-commit revert path.
