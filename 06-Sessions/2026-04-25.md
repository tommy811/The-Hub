### Key Decisions (sync 13)

- **launchd over Docker / systemd / brew services.** Native to macOS, zero deps, ships with the OS. ThrottleInterval=10s prevents tight crash loops if the worker ever has bad env vars.
- **Polling at 3s, not Supabase Realtime.** Realtime needs `ALTER PUBLICATION` per table + extra client setup. Polling at 3s is dead simple, only fires while a card is actually in `processing` state (component unmounts otherwise), latency is invisible at human pace.
- **5 progress stages, 2-3 word labels** ("Fetching profile" / "Resolving links" / "Analyzing" / "Saving" / "Done"). Per Simon's "few words, simple, don't overcrowd UI".
- **No ETA in v1.** Pipeline time varies a lot with destination count; a fake countdown would lie. The label tells you what's happening.
- **Progress writes are non-fatal.** Wrapped in try/except — if the supabase update fails, the pipeline keeps running.
- **Token rotation handled inline, not via a "rotate token" workflow.** Single line replacement in `.env` with backup; reversible if anything went wrong.
- **Three small commits per phase rather than one big batch.** 10 commits this pass. Easier `git log` reading; per-commit revert path.

**Session type:** purely conversational/diagnostic — no code committed. ### The question

Simon noticed in the live data: across all 4 creators, the only profiles we have are IG, TT, and `tapforallmylinks.com` link-in-bio rows. ### The investigation

Confirmed by reading:
- `scripts/pipeline/resolver.py:135-253` — `resolve_seed` only iterates `seed_ctx.external_urls` and `gemini_result.text_mentions` (both are seed-scoped). When a secondary IG/TT is enriched (its `InputContext` ends up in `enriched_contexts`), we have its `external_urls` and `bio` but **never feed them back into Stage B**.
- `scripts/aggregators/{linktree,beacons,custom_domain}.py` — these expand aggregator pages to children, but only get called from the seed's Stage B.
- Live DB query (`profiles` + `profile_destination_links` for all 4 creators) confirmed: every secondary social profile has its IG bio harvested for the *fetch context* but no aggregator children listed.

Earlier in the session I had also discovered that `tapforallmylinks.com/<slug>` was returning HTTP 404 from this machine — that turned out to be a network/geo issue, not a bug in our scraper. Simon confirmed the aggregator scrape works through a VPN. So the dead-aggregator hypothesis is closed; the open issue is the recursion gap.

### Confirmed: Apify cost shape

`apify/instagram-scraper` in `details` mode returns profile metadata + N most-recent posts in **one** actor run, where N = `resultsLimit`. Currently we set `resultsLimit: 1` for cheap discovery (~$0.10/handle). Profile metadata vs post scraping aren't separate API endpoints — they're parameters on the same actor. We could batch multiple handles in one run via `directUrls: [...]` for further savings.

Even so, the recommendation is to **keep discovery and post-scraping as separate jobs**:
- Discovery: triggered on add / Re-run, ~$0.10/handle, `resultsLimit: 1`
- Post-scraping (Phase 2 §14 #9): cron'd every 12h, `resultsLimit: 50-100`, ~$0.30-0.50/handle

Collapsing them would 5x the cost of every "Re-run Discovery" click and re-pull posts we already have. Split is correct.
